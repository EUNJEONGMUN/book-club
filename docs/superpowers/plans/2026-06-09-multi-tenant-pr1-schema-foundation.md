# Multi-tenant Phase A — PR 1: Schema Foundation Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Add the new multi-tenant schema (`clubs`, `club_members`, `club_invites`) and migrate existing data into a default '부글부글' club. **No application code changes** — the running app keeps working exactly as before because existing tables, columns, and RLS are untouched.

**Architecture:** Pure additive DB migration delivered as two sequential SQL files: one DDL (tables/enum/functions/RLS for new tables), one DML (default club + member backfill + meetings.club_id backfill). The TS types layer is regenerated. Existing `profiles.approved`, existing RLS, and existing `/meetings/*` routes stay in place; PR 5 cleans them up after the rest of phase A ships.

**Tech Stack:** Supabase Postgres (local docker + Cloud), Supabase CLI, TypeScript, pnpm, Vitest

**Reference spec:** `docs/superpowers/specs/2026-06-09-multi-tenant-clubs-design.md`

---

## File Structure

| File | Action | Purpose |
|------|--------|---------|
| `supabase/migrations/20260609000001_multi_tenant_schema.sql` | Create | enum, tables, indexes, helper functions, RLS for new tables, `meetings.club_id` column (nullable) |
| `supabase/migrations/20260609000002_multi_tenant_data.sql` | Create | INSERT default 부글부글 club + backfill `club_members` from `profiles.approved=true` + backfill `meetings.club_id` |
| `lib/database.types.ts` | Regenerate | Auto-generated by `supabase gen types`; commit the diff |
| `lib/types.ts` | Modify | Add type aliases for `Club`, `ClubMember`, `ClubInvite`, `ClubMemberRole` |

The two SQL files split DDL from DML so each can be re-applied independently for testing (DDL alone is idempotent in the sense that `supabase db reset` re-runs it identically; DML only runs once meaningfully).

## Task 5: Add helper SQL functions

**Files:**
- Modify: `supabase/migrations/20260609000001_multi_tenant_schema.sql` (append)

These functions are called from RLS policies (later tasks) and from server actions (later PRs). `SECURITY DEFINER` lets them bypass the caller's RLS to read `club_members` — necessary because the caller's own RLS is what we're trying to evaluate.

- [ ] **Step 1: Append the functions**

```sql

-- Helper: 현재 사용자가 특정 club의 active member(admin 또는 member)인가?
CREATE OR REPLACE FUNCTION is_club_member(target_club_id UUID)
RETURNS BOOLEAN
LANGUAGE sql STABLE SECURITY DEFINER
SET search_path = public
AS $$
  SELECT EXISTS (
    SELECT 1 FROM club_members
    WHERE club_id = target_club_id
      AND user_id = auth.uid()
      AND role IN ('admin', 'member')
  );
$$;

-- Helper: 현재 사용자가 특정 club의 admin인가?
CREATE OR REPLACE FUNCTION is_club_admin(target_club_id UUID)
RETURNS BOOLEAN
LANGUAGE sql STABLE SECURITY DEFINER
SET search_path = public
AS $$
  SELECT EXISTS (
    SELECT 1 FROM club_members
    WHERE club_id = target_club_id
      AND user_id = auth.uid()
      AND role = 'admin'
  );
$$;
```

- [ ] **Step 2: Commit**

```bash
git add supabase/migrations/20260609000001_multi_tenant_schema.sql
git commit -m "feat(schema): add is_club_member / is_club_admin helper functions"
```

## Task 7: Add RLS policies for `clubs`

```sql

-- RLS: clubs
ALTER TABLE clubs ENABLE ROW LEVEL SECURITY;

-- SELECT: 자신이 active member(admin/member)인 그룹만
CREATE POLICY clubs_select_member ON clubs
  FOR SELECT TO authenticated
  USING (is_club_member(id));

-- INSERT: 인증된 사용자 누구나. created_by는 자기 자신이어야 함
CREATE POLICY clubs_insert_self ON clubs
  FOR INSERT TO authenticated
  WITH CHECK (created_by = auth.uid());

-- UPDATE: admin만 (이름/설명 수정)
CREATE POLICY clubs_update_admin ON clubs
  FOR UPDATE TO authenticated
  USING (is_club_admin(id))
  WITH CHECK (is_club_admin(id));

-- DELETE: admin만
CREATE POLICY clubs_delete_admin ON clubs
  FOR DELETE TO authenticated
  USING (is_club_admin(id));
```

## Task 8: Add RLS policies for `club_members`

**Files:**
- Modify: `supabase/migrations/20260609000001_multi_tenant_schema.sql` (append)

- [ ] **Step 1: Append the policies**

```sql

-- RLS: club_members
ALTER TABLE club_members ENABLE ROW LEVEL SECURITY;

-- SELECT: 자신이 active member인 그룹의 모든 member row (pending 포함, admin이 신청자 보려면 필요)
CREATE POLICY club_members_select_member ON club_members
  FOR SELECT TO authenticated
  USING (is_club_member(club_id));

-- INSERT: RLS로는 금지. 모든 INSERT는 server action(SECURITY DEFINER 함수)에서만
--   - 그룹 만들기: admin row INSERT (server action에서 트랜잭션으로)
--   - 가입 신청: pending row INSERT (server action이 token 검증 후 우회)
-- 정책을 아예 만들지 않으면 RLS 활성화된 테이블은 service role만 INSERT 가능

-- UPDATE: admin만 그 그룹의 row 변경 (pending → member 승인, admin↔member 이양)
CREATE POLICY club_members_update_admin ON club_members
  FOR UPDATE TO authenticated
  USING (is_club_admin(club_id))
  WITH CHECK (is_club_admin(club_id));

-- DELETE: 본인 row 삭제 (탈퇴) OR admin이 다른 멤버 삭제
CREATE POLICY club_members_delete_self_or_admin ON club_members
  FOR DELETE TO authenticated
  USING (user_id = auth.uid() OR is_club_admin(club_id));
```

## Task 10: Create the data migration file (default 부글부글 club + member backfill)

**Files:**
- Create: `supabase/migrations/20260609000002_multi_tenant_data.sql`

This file inserts the default 부글부글 club and backfills `club_members` from existing approved profiles. The author email lookup (`scone@ignite.co.kr`) makes it portable — works in local (if seed includes that email) and production (where that account exists). For local without seed, the INSERTs simply affect zero rows, which is fine.

- [ ] **Step 1: Create the file**

```sql
-- Phase A · PR 1: data migration (default club + member backfill + meetings.club_id backfill)
-- See docs/superpowers/specs/2026-06-09-multi-tenant-clubs-design.md

-- 1. default '부글부글' 그룹 INSERT. EUNJEONGMUN이 admin
--    auth.users.email 기준으로 찾아서 portable (local에 같은 email 없으면 0 row insert)
INSERT INTO clubs (name, created_by)
SELECT '부글부글', id FROM auth.users WHERE email = 'scone@ignite.co.kr'
ON CONFLICT DO NOTHING;

-- 2. 기존 approved 사용자 모두 default 그룹의 멤버로
--    scone@ignite.co.kr은 admin, 나머지는 member
INSERT INTO club_members (club_id, user_id, role)
SELECT
  (SELECT id FROM clubs WHERE name = '부글부글' LIMIT 1),
  p.id,
  CASE WHEN u.email = 'scone@ignite.co.kr' THEN 'admin'::club_member_role
       ELSE 'member'::club_member_role END
FROM profiles p
JOIN auth.users u ON u.id = p.id
WHERE p.approved = true
  AND EXISTS (SELECT 1 FROM clubs WHERE name = '부글부글')
ON CONFLICT (club_id, user_id) DO NOTHING;

-- 3. 기존 모임 모두 default 그룹 소속으로
UPDATE meetings
SET club_id = (SELECT id FROM clubs WHERE name = '부글부글' LIMIT 1)
WHERE club_id IS NULL
  AND EXISTS (SELECT 1 FROM clubs WHERE name = '부글부글');
```

`ON CONFLICT DO NOTHING` makes the file idempotent — re-running won't error out on duplicate inserts. The `EXISTS` guard on (2) and (3) means: if step (1) had zero matching rows (no `scone@ignite.co.kr` in `auth.users` locally), steps (2) and (3) also do nothing rather than failing on NULL `club_id`.
